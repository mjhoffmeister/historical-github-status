Methodology & caveats

Source
Incidents from githubstatus.com's public API and history pages. Fetch script: scripts/fetch-data.js.
Metric
For each (component, month) we sum the duration of incidents that affected that component, in minutes. "Uptime %" is derived as 1 − minutes / (30·24·60) for display only; the underlying metric is downtime minutes.
Y-axis
The default chart starts at 0 minutes. The "Uptime %" toggle is intentionally zoomed and labeled as such — useful for SLA comparison but it visually exaggerates differences of fractions of a percent.
Scope drift
The status page tracked ~7 components in 2016 and ~11 today (Actions added 2019, Packages 2019, Codespaces 2021, Copilot 2022). A like-for-like longitudinal comparison requires holding the component set fixed — that is the Stable core default.
Neutral annotations
Material events — component launches, major postmortems, methodology changes, and corporate events including the Microsoft acquisition — are included so readers can look for correlation themselves. They are shown as dated tick marks in a single context strip, not as vertical lines splitting the chart.
Partial months
The current month is shown as a hollow point and is excluded from the rolling mean and ±1σ band, because partial-month totals would otherwise look like dramatic dips or spikes.
What this chart cannot tell you
Severity-weighted user impact, regional variation, performance degradations that did not trip an incident, or how status-page reporting practices have themselves changed over time.
Data freshness
The live API only exposes the last ~50 incidents. Older months come from a committed snapshot derived from publicly-documented incidents; rerun the fetcher to refresh recent months.
